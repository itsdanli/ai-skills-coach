Day 2 — Prompt Caching Breakpoints

By the end of this you'll be able to make a batch API script cheaper and faster by caching the static prefix — and know the one ordering mistake that quietly disables it.

Why this matters

You wire up a script that classifies 4,000 support tickets, each call carrying the same 18k-token schema and instruction block in front of one short ticket. It runs, it's slow, and the bill is brutal — because you re-sent that identical prefix 4,000 times. Worse: if you glue the changing ticket text before the static schema in one big string, the cached prefix never matches from one call to the next, so you pay the write premium on every request and get zero hits. The technique is only worth it if the cached part is byte-for-byte identical across calls, and one careless concatenation breaks that.

The mechanic

Prompt caching lets Claude store a prefix of your request and reuse it. You mark the end of the cacheable region with cache_control: {"type": "ephemeral"} on a content block. The cache spans the whole prompt in a fixed order — tools, then system, then messages — up to and including that block. Everything before the breakpoint is cached; everything after stays dynamic. You get up to four breakpoints per request; a hit requires the breakpoint and everything before it to be identical. Default TTL is five minutes, refreshed on every read, so a steady stream of calls keeps the prefix hot indefinitely (a 1-hour tier exists at extra cost).

import anthropic
client = anthropic.Anthropic()

SCHEMA = open("340b_table_schema.md").read()   # ~18k tokens, identical every call

resp = client.messages.create(
    model="claude-sonnet-5",
    max_tokens=512,
    system=[
        {"type": "text", "text": "Classify each Plenful ticket by workflow area."},
        {
            "type": "text",
            "text": SCHEMA,
            "cache_control": {"type": "ephemeral"}   # <-- breakpoint: everything up to here is cached
        },
    ],
    messages=[
        {"role": "user", "content": ticket_text},    # dynamic, comes AFTER the breakpoint
    ],
)
print(resp.usage.cache_creation_input_tokens,      # > 0 only on the first (write) call
      resp.usage.cache_read_input_tokens)          # > 0 on every hit afterward

The breakpoint sits on the last static block — the schema — not on the instruction line above it, because caching includes everything up to the tagged block. The ticket lives in messages, after the cached region, so it varies freely without disturbing the prefix. The two usage counters are your proof: the first call reports cache_creation_input_tokens (you paid ~25% over base to write), and every call after reports cache_read_input_tokens billed at roughly a tenth of the base input price.

The upgrade over the naive version

Before: system = INSTRUCTIONS + SCHEMA + "\n\nTicket: " + ticket_text — one string, no cache_control. Because the ticket changes every call and sits inside the prefix, there is nothing stable to cache; you re-send and re-pay for 18k tokens 4,000 times.

After: the static schema is its own block tagged with cache_control, and the ticket moved into messages behind it. Call one writes the prefix; calls two through 4,000 read it. Same output, a fraction of the cost, and lower latency because Claude skips re-processing the prefix. The only change was where the changing text lives relative to the breakpoint.

PHI note: keep the cached prefix to non-PHI schema and instructions, and put the per-record content — which may contain PHI — after the breakpoint, sending only the minimum each call needs. Caching a stable prefix and minimizing dynamic PHI pull in the same direction.

Try it now (10 minutes)

Take any script where you call the API in a loop with a big fixed system prompt (a classifier, a transcript-to-Linear extractor, a PQL generator that ships the table schema). Split system into a list of two blocks — instructions, then the large static context — and put cache_control: {"type": "ephemeral"} on the second. Run the loop twice over the same two records and print resp.usage.cache_creation_input_tokens and resp.usage.cache_read_input_tokens. Success looks like: creation tokens are non-zero only on the first call, and read tokens are non-zero (≈ your prefix size) on every call after. If reads stay at zero, something dynamic slipped in front of the breakpoint — find it.

Apply it to my workflow

Batch ticket triage: the workflow-classification prompt and taxonomy are identical across every ticket in a run — cache them once, pay per-ticket only for the ticket. Transcript-to-Linear extraction: the extraction rubric and team/owner conventions are the stable prefix; the transcript is the dynamic tail. Customer-environment Q&A scripts: the 340B schema reference and query-style guide are a fixed prefix you can keep hot across a whole interactive session, so each follow-up question only bills for the question.

One-line bookmark

Put everything that never changes first, drop one cache_control breakpoint after it, and keep the varying text strictly behind that line.
